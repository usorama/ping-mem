### GAP-C1: LLMEntityExtractor is fully coded but never instantiated in production

**Verified claim**: `createRuntimeServices()` in `/Users/umasankr/Projects/ping-mem/src/config/runtime.ts`
returns a `RuntimeServices` object that has no `llmEntityExtractor` field at all — the type doesn't
include it. Neither `runtime.ts` nor any server startup path instantiates `LLMEntityExtractor`.

The guard at `ContextToolModule.ts` line 443:
```ts
if (useLlmExtraction && this.state.llmEntityExtractor) {
```
...is permanently false. LLM entity extraction is DEAD CODE.

**Impact on user capability**: Every `context_save` call skips LLM-powered entity extraction.
Memories are stored as raw strings with no structured entities/relationships. Knowledge graph
enrichment from memory saves is completely absent. The feature exists in tests but not in
production.

**Fix**: In `/Users/umasankr/Projects/ping-mem/src/config/runtime.ts`:
1. Import `LLMEntityExtractor` from `../graph/LLMEntityExtractor.js`
2. Add `llmEntityExtractor?: LLMEntityExtractor` to `RuntimeServices`
3. At end of `createRuntimeServices()`, check `OPENAI_API_KEY` env var and if present:
   ```ts
   services.llmEntityExtractor = new LLMEntityExtractor({ openai: new OpenAI({ apiKey }) });
   ```
4. Thread the instance through `rest-server.ts` and `PingMemServer.ts` startup configs.

## HIGH GAPS (degrade capability)

### GAP-H1: OLLAMA_URL and OLLAMA_EMBED_MODEL absent from docker-compose.prod.yml

**Verified claim**: `/Users/umasankr/Projects/ping-mem/docker-compose.prod.yml` has no `OLLAMA_URL`
or `OLLAMA_EMBED_MODEL` env vars. The dev compose (`docker-compose.yml`) has both:

```
- OLLAMA_EMBED_MODEL=nomic-embed-text
```

file on VPS).

**Impact on user capability**: Production embeddings fall through the entire Ollama→Gemini→OpenAI
fallback chain. If `.env` on the VPS lacks API keys, production falls to keyword-only BM25 search
with NO semantic similarity. This silently degrades `context_auto_recall`, `context_hybrid_search`,
and codebase semantic search without any error surfaced to users.

**Fix**: Add to `/Users/umasankr/Projects/ping-mem/docker-compose.prod.yml` under `ping-mem` service:
```yaml
- OLLAMA_URL=${OLLAMA_URL:-}
- OLLAMA_EMBED_MODEL=${OLLAMA_EMBED_MODEL:-nomic-embed-text}
- GEMINI_API_KEY=${GEMINI_API_KEY:-}
- OPENAI_API_KEY=${OPENAI_API_KEY:-}
```
Prod VPS likely won't have Ollama, so at minimum the API key vars must be explicit so operators
know they must be set in `.env`.

### GAP-H2: TranscriptMiner has no EventStore wiring — mining completions are unobservable

**Verified claim**: `/Users/umasankr/Projects/ping-mem/src/mining/TranscriptMiner.ts` lines 357-360
contain a literal TODO comment:
```ts
// EventStore dependency. Wire it in via constructor injection and call:
//   this.eventStore.createEvent(sessionId, "TRANSCRIPT_MINED", { ... })
// This requires a constructor change and passing the EventStore from rest-server.ts.
```
`DreamingEngine` DOES emit `INSIGHT_DERIVED` events (line 180, 210, 441 — verified). Only
TranscriptMiner is missing the wiring.

**Impact on user capability**: Transcript mining completions are invisible to the event system.
No TRANSCRIPT_MINED events means: no webhook triggers, no audit trail, no dashboard visibility,
no downstream consumers (like consolidation) can react to new transcripts being mined.

**Fix**: Constructor-inject `EventStore` into `TranscriptMiner`, emit `TRANSCRIPT_MINED` after
each successful mine. Pass `EventStore` from `rest-server.ts` at construction time.

### GAP-H3: Feedback loop is observational-only — poor recall signals don't drive real-time correction

**Verified claim (from Agent 6 report, not independently re-verified against specific lines)**:
System records `MEMORY_RECALLED` events and tracks FSRS relevance decay, but consolidation
that could improve memory quality only runs on a schedule — not triggered by failed/low-score
recalls.

**Impact on user capability**: When `context_auto_recall` returns low-confidence results,
nothing self-corrects. The system accumulates evidence of recall failures but doesn't act on
it between scheduled maintenance windows. Memory quality degrades silently until the next
scheduled consolidation run.

**Fix**: Add a "poor recall" threshold check in `handleAutoRecall`. When `filtered.length === 0`
or all scores are below 0.3, emit a `RECALL_MISS` event. Add a lightweight consolidation
trigger that fires after N consecutive misses for a given query pattern.

## MEDIUM GAPS (reduce quality, not blocking)

### GAP-M1: DreamingEngine uses callClaude() (Claude API direct) not LLMProxy/Ollama

**Verified claim**: DreamingEngine imports `callClaude` from `../llm/ClaudeCli.js` (line 25) and
calls it at lines 267 and 300. No reference to `llmProxy`, `LLMProxy`, or `ollama` anywhere in
the file.

**Impact**: Dreaming always requires Claude API access — it cannot fall back to Ollama. In
environments where only Ollama is available (or Claude API is rate-limited), dreaming silently
fails. Also means dreaming has a separate, unmonitored LLM path outside the LLMProxy fallback
chain.

**Fix (low priority)**: Route DreamingEngine through LLMProxy so it participates in the
Ollama→Gemini→Claude fallback chain. Or document that dreaming requires Claude API and surface
a clear error when unavailable.

---

### GAP-M2: proxy-cli.ts is the recommended transport, but Codex/direct consumers still use cli.js

**Verified claim (from Agent 1, not re-verified in codebase)**: Some consumers use direct `cli.js`
mode which opens SQLite directly, causing concurrent access conflicts when Docker container also
has the DB open.

**Impact**: Intermittent SQLite lock errors for consumers using direct mode. Not a blocker since
the proxy mode works, but a latent reliability issue.

**Fix**: Audit all external consumer configurations. Where `cli.js` is referenced, replace with
`proxy-cli.js` + `PING_MEM_REST_URL=http://localhost:3003`.

### GAP-M3: docker-compose.prod.yml port binding is 3003, not 3000

**Verified claim**: `/Users/umasankr/Projects/ping-mem/docker-compose.prod.yml` line 48 shows:
```yaml
ports:
  - "127.0.0.1:3003:3003"
```
But per MEMORY.md, the VPS production deployment uses port 3000 internally (with Nginx proxying
from external). The MEMORY.md notes: "must `sed` docker-compose.prod.yml to restore port 3000
after rsync."

**Impact**: A manual sed step is required every deploy cycle. If forgotten, Nginx proxy breaks.
This is a deploy-time footgun, not a runtime gap when properly deployed.

**Fix**: Either maintain a separate `docker-compose.prod.yml` on VPS that is not overwritten by
rsync, or add a deploy script that patches the port automatically. Document the sed step
prominently in deploy runbook.
