## Findings

### F1: Knowledge FTS5 Multi-Word Queries Return 0 Results
- **Severity**: Medium
- **Query**: `{"query": "biometric authentication android"}` → 0 results
- **Query**: `{"query": "Biometric"}` → 1 result (correct)
- **Root cause**: FTS5 multi-word queries are treated as phrase search in SQLite. Space-separated terms require explicit `OR` or `*` syntax.
- **Workaround**: Use single-word queries until FTS5 tokenizer is configured for multi-term OR queries.

### F2: Session/Context API Not on REST Server
- **Severity**: Low (by design)
- **Behavior**: `POST /session/start` → 404 on port 3003
- **Root cause**: Session endpoints are on the MCP/SSE server (port 3000), not the REST server. This is intentional architecture.
- **Impact**: REST clients cannot create sessions without knowing the MCP server port.
- **Workaround**: Use port 3000 for session operations; document this in API reference.

### F3: Deterministic Search Scores Are Low (< 0.20)
- **Severity**: Low (expected behavior)
- **Max score observed**: 0.2036 (streak-related query)
- **Root cause**: Deterministic n-gram hash vectors produce low cosine similarity when query terms don't exactly overlap with chunk n-grams. L2 normalization distributes similarity across many dimensions.
- **Impact**: Consumers may implement a minimum score threshold (e.g., 0.15) and miss relevant chunks.
- **Recommendation**: Document expected score range (0.05-0.25 for deterministic, 0.5-0.9 for ML embeddings) in API docs.
